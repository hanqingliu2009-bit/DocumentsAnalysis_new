# 后续工作：多模态嵌入与文档内嵌图（当前仅文字）

本文记录**暂不实现**、但已与产品/技术路径相关的待办，避免与「豆包 embedding-vision + `/embeddings/multimodal`」的当前能力混淆。

---

## 当前约定（已实现范围）

| 项 | 说明 |
|----|------|
| 嵌入后端 | `EMBEDDING_BACKEND=volcengine` 且 `EMBEDDING_USE_MULTIMODAL_API=true` 时，走方舟 **`POST .../embeddings/multimodal`**。 |
| 索引与查询 | **仅向 API 传入文本块** `{"type":"text","text":"..."}`，不把文档中的图片送入模型。 |
| 元数据扩展（已实现、可选使用） | 若在 `DocumentChunk.metadata` 中写入 **`embedding_image_urls`** / **`image_urls`**（公网可访问 URL），`EmbeddingGenerator.embed_chunks` 会把图文块一并拼进 `input`。当前流水线**不会自动写入**这些字段。 |

结论：**暂时只考虑文字**；与「复制粘贴进 Word 的内嵌图」相关的向量化与检索，**列为后续开发**。

---

## 现状缺口（与「粘贴图」直接相关）

| 来源 | 现状 | 影响 |
|------|------|------|
| **DOCX** | `processor._extract_docx` 仅汇总 **`paragraphs` 文本**，未遍历 `runs` / 内联形状，**不抽取嵌入图**。 | 粘贴图不参与分块与嵌入。 |
| **PDF** | `_extract_pdf` 以文本提取为主（如 pypdf），**未做版面/图片抽取**。 | 扫描件或内嵌位图通常无有效文本时，检索价值低；图亦未进多模态。 |
| **公网图 URL** | 仅当业务侧写入 `metadata` 中的 URL 时才会随 chunk 请求多模态接口。 | 内网-only、纯粘贴图无 URL 时，当前链路用不上图。 |

## 待开发内容（建议按优先级）

1. **需求边界**  
   - 明确优先格式：**DOCX 内联图** / **PDF 内嵌图** / 二者都要。  
   - 明确查询侧：用户是否以**纯文本**提问为主（跨模态检索是否必须「以文搜图」）。

2. **从文档抽取图片字节**  
   - DOCX：`python-docx` 与底层 `document.part.related_parts` / `blip` 等，或拆 `word/media`（zip）定位 `r:embed` 与图片关系。  
   - PDF：按页或按 XObject 导出图像（需评估依赖：pypdf / pdf2image / 其他）。

3. **与方舟多模态 API 的对接方式**（二选一或组合）  
   - **方案 A**：若官方支持 **`image_base64`**（或 `image_url.url` 使用 `data:image/png;base64,...`），则从内存直接编码上传，**无需公网托管**。  
   - **方案 B**：仅支持 HTTP(S) 拉图时，将抽取结果**短时上传**至对象存储（如 TOS），使用**带签名的临时 URL** 填入 `embedding_image_urls`（非永久公网暴露）。

4. **分块与向量对齐策略**  
   - 同一 chunk：**一段文字 + N 张相关图** 拼成一条 `input`（注意 token/大小限制与失败重试）。  
   - 或：**图单独成 chunk**（metadata 关联前后文本 chunk id），检索时再合并上下文。需与 `CHROMADB_COLLECTION` 维度及 `SIMILARITY_THRESHOLD` 行为一并验证。

5. **工程与运维**  
   - 大图压缩、格式统一（PNG/WebP）、单文档图片数量上限、超时与错误日志。  
   - 隐私：抽取图是否落盘、保留多久、是否与 `FILE_STORAGE_PATH` 策略一致。
