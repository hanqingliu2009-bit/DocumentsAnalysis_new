## 2. 数据层：合并 JSON（阶段 A）

### 2.1 新增脚本 `backend/scripts/merge_splite_json.py`

- **作用**：扫描目录下 `*.json`（排除输出文件名 `merged_splite_corpus.json`），校验每文件根对象含 `document_id`、`document_name`、`chunks[]`，写出统一语料结构。
- **输出结构**：根级 `version`、`generated_at`（UTC ISO8601）、`source_count`、`chunk_count`、`sources[]`；每个 source 含原文件名 `file` 及上述字段与 `chunks` 数组。
- **入口**：`python backend/scripts/merge_splite_json.py`，可选 `--input` / `--output`（路径相对仓库根解析）。

### 2.2 数据产物 `files/merged_splite_corpus.json`

- 由脚本生成并纳入版本库（体积较大）：当前示例为 **7 个文档、651 条 chunk**。
- **注意**：后续若 `files/` 下源 JSON 变更，需重跑合并脚本并视情况更新该产物与 Chroma 索引。

## 3. 向量层：Splite + BGE 灌库（阶段 B）

### 3.1 新增脚本 `backend/scripts/ingest_merged_corpus_bge.py`

- **流程**：
  1. **`os.chdir(backend/)`**：保证 `config.settings` 中 `./data` 类路径落在 **`backend/data/`**，与从 `backend` 启动的 `uvicorn` 一致，避免在仓库根误建 `data/`。
  2. 在 **import `config` 之前** 设置 `os.environ["EMBEDDING_BACKEND"]="local"`、`EMBEDDING_MODEL`、`EMBEDDING_DIMENSION`（默认 `BAAI/bge-large-zh-v1.5` / **1024**），使灌库不依赖 `.env` 里火山嵌入。
  3. 读取合并语料，将每条 chunk 编成检索用文本：`【group_title】\ncontent`（无标题则只用正文）。
  4. 使用 **`DocumentChunk`** + **`VectorStore.add_chunks`** 写入 **`HYBRID_SPLITE_COLLECTION` 默认名**对应的 Chroma 集合（默认 **`splite_bge_zh_v15`**）。
- **Chunk ID**：`{document_id}:{chunk_id}`，避免跨文档 `chunk_id` 冲突。
- **CLI**：`--recreate` 时先 `delete_collection` 再 `get_or_create`；`--batch-size`、`--corpus`、`--collection`、`--model`、`--embedding-dimension` 可覆盖。

### 3.2 `backend/storage/vector_store.py`：`add_chunks` 元数据合并

原先写入 Chroma 的 metadata 仅含 `document_id`、`chunk_index`、`start_char`、`end_char`。现把 **`DocumentChunk.metadata`** 中 **标量**（`str|int|float|bool`）合并进同一条 metadata；非标量转为 **截断字符串**（≤500 字符），以满足 Chroma 对 metadata 类型的限制。

灌库脚本为每条 chunk 写入的 metadata 包括：`document_name`、`source_file`、`group_title`、`split_chunk_id` 等，便于检索结果展示与溯源。

### 3.3 `backend/.env.example`（Splite / Ops）

- 增加 Splite 与 Hybrid 的注释示例块。
- 增加 **Ops** 小节：合并命令、灌库命令、HF 缓存说明、`backend/data/vector_db`、重灌与 **`POST /api/splite_search`** 调试提示。

### 3.4 `.gitignore`

- 增加仓库根 **`/data/`**，防止在错误 cwd 下运行脚本时产生的孤立目录被误提交（`backend/data/` 仍按原规则忽略）。

## 4. RAG 核心：`RAG_BACKEND=hybrid`（阶段 C）

### 4.1 `backend/config.py`

新增配置项（均可被环境变量覆盖）：

| 字段 | 含义 |
|------|------|
| `RAG_BACKEND` 取值 `hybrid` | 启用混合检索分支（注释中说明三种 backend 语义） |
| `HYBRID_SPLITE_COLLECTION` | Splite 专用 Chroma 集合名，默认 `splite_bge_zh_v15` |
| `HYBRID_SPLITE_EMBEDDING_MODEL` | Splite 查询与灌库一致的 HF 模型 id |
| `HYBRID_VECTOR_TOP_K` | 向量路取回条数默认上限 |
| `HYBRID_CONTEXT_MAX_CHARS` | 合并后送入 LLM 的上下文硬截断长度 |

### 4.2 `backend/core/rag.py`

**分流**：在 `query()` 中，对 `settings.RAG_BACKEND` 判断时，**先于** `external_graph` 处理 **`hybrid`**：

```62:67:backend/core/rag.py
        try:
            backend = (getattr(settings, "RAG_BACKEND", "chromadb") or "chromadb").strip().lower()
            if backend == "hybrid":
                return self._query_hybrid(question, document_ids, top_k, retrieval_query)
            if backend == "external_graph":
                return self._query_external_graph(question, retrieval_query)
```

**`_query_hybrid` 要点**：

- 使用 **`concurrent.futures.ThreadPoolExecutor(max_workers=2)`** 并行执行：
  - **图路**：`fetch_graph_context(q)`，`q` 为 `retrieval_query` 或整段 `question` 的去空白结果（与 chat 行为一致）。
  - **向量路**：`EmbeddingGenerator(model_name=HYBRID_SPLITE_EMBEDDING_MODEL)` + **`embed_texts(..., embedding_backend="local")`**，再 **`VectorStore(collection_name=HYBRID_SPLITE_COLLECTION).search`**，`top_k` 取 `HYBRID_VECTOR_TOP_K` 或请求参数或 `TOP_K_RETRIEVAL`。
- **单侧失败**：异常时该路结果为空，另一路仍可参与；两路皆空时走与图模式类似的 **无命中 + LLM / 系统提示** 分支。
- **上下文拼装**：`_build_hybrid_context` → `### 图数据库检索结果` + `### 本地手册向量检索` + 复用 `_build_context` 对向量命中列表排版；再经 **`_truncate_hybrid_context`** 按 `HYBRID_CONTEXT_MAX_CHARS` 截断。
- **LLM**：`_generate_answer_hybrid` 使用新的 system 说明（两路来源、引用习惯、语言与用户一致）。
- **返回**：`answer_mode="hybrid_graph_vector"`；`sources` 中图侧条目附加 **`source_channel":"graph"`**，向量侧由 **`_format_sources(..., source_channel="vector")`** 写入。

**`_format_sources` 扩展**：增加可选关键字参数 **`source_channel`**，在字典中写入该字段（供 API 层 `SourceCitation` 使用）。

### 4.3 `backend/storage/vector_store.py`：`embed_texts` / `embed_text`

### 4.4 `backend/main.py`：启动预热

- 当 **`RAG_BACKEND=hybrid`**：在 lifespan 中 **只预热 Splite BGE**（`EmbeddingGenerator(model_name=...).embed_texts(["warmup"], embedding_backend="local")`），不再走默认 `EmbeddingGenerator().embed_text("warmup")` 的火山路径，避免无嵌入配置时无意义失败。
- **`external_graph`** 行为保持：跳过默认嵌入预热。

### 4.5 `backend/api/query.py`

- **`SourceCitation`**：新增可选字段 **`source_channel`**（`graph` / `vector`）。
- **`QueryResponse` / `ChatResponse`**：`answer_mode` 的 OpenAPI 描述中增加 **`hybrid_graph_vector`**。

## 6. 行为与兼容性说明

- **`chromadb`**、**`external_graph`** 原有分支 **未删除**；仅新增 **`hybrid`** 分支与相关配置、脚本、接口。
- **默认 `main.vector_store`** 仍指向 **`CHROMADB_COLLECTION`**（文档上传索引）；**Splite** 使用 **独立 `VectorStore` 实例 + 独立集合名**，避免与默认集合 **向量维度混用**。
- **Hybrid 向量路** 与 **全局 `EMBEDDING_BACKEND`** 解耦：依赖代码内 **`embedding_backend="local"`** + **`HYBRID_SPLITE_EMBEDDING_MODEL`**，便于生产环境主链路继续用火山嵌入而 Splite 仍用 BGE。
