## 事项一：将 Hugging Face 嵌入模型拉到本地（离线可用）

### 目标

- 当前 `EmbeddingGenerator`（`backend/storage/vector_store.py`）通过 **`SentenceTransformer(settings.EMBEDDING_MODEL)`** 加载默认 **`sentence-transformers/all-MiniLM-L6-v2`**，首次运行会从 **Hugging Face Hub** 拉权重，依赖网络。
- 目标：**在内网/无公网环境下仍能加载同一套权重**，并尽量固定缓存位置，便于备份与部署。

### 现状摘要

| 位置 | 说明 |
|------|------|
| `config.py` | `EMBEDDING_MODEL`、`EMBEDDING_DIMENSION`（384）；另有 **`EMBEDDING_CACHE_DIR`**、**`TRANSFORMERS_OFFLINE`**（并导出离线相关 env）。 |
| `vector_store.py` | 仍为 **`SentenceTransformer(self.model_name)`**（Hub id 或将来本地路径）；缓存依赖 **`HF_HOME`**（由 `EMBEDDING_CACHE_DIR` 在配置阶段设置）。**检索**侧已用 **`SIMILARITY_THRESHOLD`** 过滤。 |
| 依赖 | `sentence-transformers` → 底层仍用 `transformers`/`huggingface_hub` 的缓存机制 |

### 推荐方案（组合使用）

**方案 A — 仅指定本地缓存目录（改动小）**

1. 在 `config.py` 增加可选配置项，例如：  
   - `HF_HOME` 或专用 **`EMBEDDING_CACHE_DIR`**（指向 `backend/data/models/hf` 等仓库内或部署机上的目录）。  
2. 在应用启动最早阶段（如 `EmbeddingGenerator` 首次 import 前）或 `main.py` lifespan 里，对 `os.environ` 设置 **`HF_HOME`** / **`TRANSFORMERS_CACHE`**（与团队约定一个即可，避免重复）。  
3. 在**有网络**的机器上预先执行一次加载（或脚本 `python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('sentence-transformers/all-MiniLM-L6-v2')"`），把缓存目录**打包复制**到离线环境同一路径。  
4. 离线机设置 **`TRANSFORMERS_OFFLINE=1`**（或 `HF_HUB_OFFLINE=1`，视版本而定），避免误连网。

**方案 B — 使用本地文件夹路径作为模型名（推荐与 A 二选一或并存）**

1. 用 `huggingface_hub.snapshot_download` 或浏览器从 HF 页面下载模型仓库到固定目录，例如 **`backend/data/models/all-MiniLM-L6-v2/`**（目录内含 `config.json`、`model.safetensors` 等）。  
2. 将 `EMBEDDING_MODEL` 配置为该**绝对路径或相对 `backend/` 的路径**；`SentenceTransformer(本地路径)` 可直接从磁盘加载，**不依赖 Hub 元数据拉取**（仍可能读本地 `config.json`）。  
3. `.env` / `config.py` 默认值可改为该路径；文档中说明「克隆仓库后需自备模型目录或首次联网预下载」。

**方案 C — 镜像与 CI（可选）**

- 企业内网部署 **HF 镜像**（如官方文档中的镜像站），通过环境变量把 Hub 根 URL 指到镜像；适合不便拷贝大文件但可访问内网镜像的场景。

### 与现有向量数据的关系

- **权重相同、维度仍为 384** 时，已有 Chroma 里的向量与查询向量仍可比，一般**无需**因「改成本地路径」而全量重索引。  
- 若将来**更换模型**（维度或语义空间变了），需要：**重新嵌入所有 chunk 并重建/覆盖集合**，单独作为迁移任务文档化。

## 事项二：向量库无命中时走在线 LLM（Chat 与 `/api/query` 一致）

### 目标

- 当检索结果为空（相似度低于阈值导致无 chunk 进入 `retrieved`）时，在已配置 Key/模型的情况下**直接调用已配置的火山方舟（OpenAI 兼容）接口**生成回答，**不再返回**本地长段预设文案。  
- 与文档相关且**有命中**时，行为保持 **RAG + 引用片段** 不变。

### 现状摘要

| 位置 | 说明 |
|------|------|
| `core/rag.py` | 无检索命中且已配 Key/模型时**统一**调用 **`_generate_answer_without_context`**（system 含索引统计等）。未配置时仅返回需配置 **`.env`** 的短提示。 |
| `api/query.py` | **`/api/chat`** 与 **`/api/query`** 均调用 **`get_rag_pipeline().query(...)`**，无额外布尔开关区分无命中行为。 |

### 推荐方案

**核心逻辑（在 `RAGPipeline.query` 或单独抽出 `chat_with_optional_rag`）**

1. **检索阶段不变**：仍对 `retrieval_query` 做 embedding + `vector_store.search`。  
2. **分支**：  
   - **有 `retrieved`**：保持现有 `_build_context` + `_generate_answer(question, context)`（强约束「依据上下文」的 system prompt）。  
   - **无 `retrieved`**：若已配置 `VOLCENGINE_API_KEY` 与 `LLM_MODEL`，则调用 **`_generate_answer_without_context(question)`**（新方法）：  
     - `messages`：system 说明「当前知识库未检索到相关文档，请作为通用助手回答；不要编造用户拥有某份文件」。  
     - user：`question`（含 chat 历史拼接后的 `full_question` 亦可）。  
     - 返回 `sources=[]`，`confidence=0`，`context_used=0`。  
3. **保留简短引导**（可选）：在 system 或首条 assistant 中提示「如需结合已上传文档，请在 Documents 中上传并提问与文档相关的问题」。  
4. **仍无 API Key / 无模型名**：保持现有错误/提示文案，不调 LLM。

**可选增强（后续迭代）**

- **简单意图路由**：用规则或小模型判断「是否像文档问答」（例如含「根据文档」「第几章」等）再决定是否 fallback，减少无文档时的乱答（实现成本高，可二期）。  
- **若需「仅 Chat 无命中走 LLM、Query 无命中固定拒答」**：当前代码已对齐为两端一致；若要分叉，需在 `api/query.py` / `rag.py` 另行增加显式开关或路由逻辑（仓库默认不提供）。

### 风险与约定

- **成本**：无检索也会消耗 token，需在 README 或配置里说明。  
- **幻觉**：须用 system prompt 明确「无文档时不要假装引用了上传文件」。  
- **与打招呼**：已无 `_no_context_answer` 本地模板；问候语同样消耗 LLM token（若已配 Key/模型）。
